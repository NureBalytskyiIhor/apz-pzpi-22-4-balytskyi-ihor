МЕТА РОБОТИ


Основною метою цього лабораторного заняття є доопрацювання документа Vision and Scope для програмного забезпечення, яке призначене для організації, модерації та підтримки благодійних ініціатив і збору коштів.


ХІД РОБОТИ
У процесі виконання роботи було проаналізовано проблеми у сфері збору благодійних коштів, зокрема — відсутність єдиної централізованої платформи, складність перевірки ініціатив, низьку прозорість процесів для донорів та складність організації комунікації між учасниками. На основі проведеного аналізу було сформовано концепцію багатокомпонентної програмної системи, що включає: мобільний застосунок для звичайних та верифікованих користувачів; серверну частину (Node.js + MongoDB), яка відповідає за збереження даних та обробку запитів; веб-інтерфейс з можливостями адміністратора.


Мобільний застосунок призначений для звичайних та верифікованих користувачів, забезпечує перегляд зборів, створення пожертвувань, ведення історії, створення заявок на нові збори та публікацію новин.
Серверна частина виконує роль центрального обробника запитів, зберігає дані у MongoDB, реалізує авторизацію та надає REST API для клієнтів. 
Веб-додаток забезпечуватиме адміністративне управління системою, модерацію зборів, верифікацію користувачів та перегляд аналітики.


Висновки
У результаті виконання цієї лабораторної роботи мені вдалося допрацювати документ Vision & Scope для програмної системи, що призначена для організації благодійних ініціатив та збору коштів. Робота над документом допомогла мені краще усвідомити, наскільки важливо на початковому етапі розробки чітко визначити архітектуру системи, її складові та можливі обмеження.
Описані рамки початкового та майбутніх випусків дозволили сформувати цілісний план розвитку продукту. Також я зрозумів, як важливо відокремлювати ключову функціональність від другорядної, щоб зосередитися на стабільній і працюючій першій версії.

VISION AND SCOPE DOCUMENT
3.1 Scope of Initial Release (Рамки первинного випуску)


Перший реліз програмної системи для організації благодійних ініціатив включає мобільний застосунок, веб-додаток адміністратора, серверну частину та IoT-пристрій для генерації QR-кодів.
* Мобільний застосунок (Android, Kotlin + Jetpack Compose)  забезпечує можливість реєстрації, логіну, перегляду актуальних зборів, їх пошуку та сортування, перегляду деталей зборів, здійснення пожертв, перегляду особистої історії донатів, редагування профілю. Верифіковані користувачі також можуть створювати власні збори, вести та постійно оновлювати блог новин.
* Серверна частина (Node.js + Express.js + MongoDB)  відповідає за обробку REST-запитів, зберігання та обробку даних про користувачів, збори, новини та історії пожертвувань. Передбачена базова обробка помилок, проста система верифікації та автентифікації.
* IoT-пристрій для генерації QR-кодів  дозволяє створювати фізичні інтерактивні точки збору: згенерований QR-код веде на сторінку збору в мобільному застосунку, де можна миттєво задонатити. Пристрій може використовуватись на благодійних заходах чи в закладах.
* Веб-додаток адміністратора (React + REST API)  реалізовано базовий функціонал: перегляд усіх зборів, перегляд користувачів, додана базова модерація заявок на створення зборів. Основна ціль це надати зручний спосіб роботи з активністю системи для адміністратора.


3.2 Scope of Subsequent Releases (Рамки наступних випусків)


У майбутніх версіях передбачається розширення функціональності системи:
* Інтеграція платіжних систем, а саме підключення сервісів (наприклад, LiqPay, WayForPay, Portmone) для здійснення реальних транзакцій прямо в додатку.
* Автоматизована верифікація через "Дію". Реалізація перевірки особистості через API державного сервісу з метою офіційної перевірки осіб, які створюють збори.
* Адміністративна модерація.Створення повноцінної системи перевірки нових зборів, включаючи підтвердження документів, статус верифікації, виявлення фейкових активностей. Також буде необхідно створення юридичних документів для осіб, які хочуть створити збір, ці особи мають розуміти, що будуть нести кримінальну відповідальність в разу порушення правил та зловживанням набутими коштами, для цього вони будуть підписувати відповідні документи за порушення правил яких, будуть притягнуті до відповідальності.
* Система коментарів - додавання можливості залишати відгуки або питання до зборів від користувачів.
* Система підтримки (чат/звернення) - впровадження каналу зв’язку для користувачів з адміністраторами через мобільний або веб-додаток.
* Підтримка push-сповіщень - інформування користувача про статус збору, надходження новин, нагадування про пожертви.
* Панель аналітики - виведення динаміки зборів, популярності, середньої суми донату, активності користувачів.
* Мобільна версія для iOS - реалізація альтернативного клієнта для користувачів пристроїв Apple.

3.3 Limitations and Exclusions (Обмеження та виключення)


   * Проєкт немає реальної обробки платежів - усі транзакції в MVP-версії симулюються; повна інтеграція з банківськими API відсутня. 
   * Верифікація користувачів базова - лише за прапорцем isVerified, без перевірки документів чи інтеграції з "Дією".
   * Без складної модерації контенту — всі збори зʼявляються автоматично без перевірки адміністратором.
   * Без резервного копіювання БД - не реалізовано механізм автоматичного бекапу або відновлення даних.
   * Немає гейміфікації - рейтинги, значки, досягнення для користувачів не реалізовані на цьому етапі.
   * Відсутність підтримки офлайн-режиму — всі дії потребують підключення до Інтернету. Хоча для здійснення покупок, наприклад через функцію NFC, інтернет не потребується. Тому реалізувати деякий функціонал(напр. здійснення пожертвування) додатка в офлайн режимі цілком можливо.
